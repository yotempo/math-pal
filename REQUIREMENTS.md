## 二、設計約束（架構層級的硬性要求）

### A. 記憶功能的實作（對應需求 3）

**要求**：不把 Elena 所有的歷史錯題直接塞進每次的 Prompt（Token 爆炸且超出 context 限制）。正確做法：把錯題標籤（例如 #分數乘法、#追趕問題）和積分存進 Oracle Cloud 的資料庫；每次出題時由後台撈出弱項標籤，再丟給 AI。

**實作**：
- 每筆作答存入 `attempts` 表，帶 `topic` 標籤（= 錯題標籤）、難度、對錯、第幾次作答
- AI prompt 只收「聚合後的標籤摘要」：`studentContext()` 用 SQL 對近 30 天首次作答做 `GROUP BY topic` 統計，只把「弱項主題（正確率 <70%）、強項主題（≥90%）」這幾行文字放進 prompt——**原始錯題永遠不進 prompt**
- 聊天歷史上限 16 則，AI 回覆上限 500 tokens
- 出題同樣由後台標籤驅動：`targetDifficulty(topic)` 讀近 20 筆該標籤的正確率自動調難度；應用題優先出「答對次數最少」的題
- Phase 2 的「AI 動態出題」也走同一條路：後台撈弱項標籤 → 給 AI 出題指令 → 生成題目入庫，不餵歷史記錄

### B. 避免 AI 算錯數學（對應需求 10）

**要求**：LLM 算數學有時會翻車。批改答案一律以題庫的標準答案（correct_answer）用後台程式碼做精確比對，AI 不參與判分。
（2026-07-08 修訂：AI 是「學伴」，隨時可以發問——不設「錯兩次才解鎖」的限制；判分歸程式、引導歸 AI 的分工不變。）

**實作**：
- **批改 100% 由程式碼完成**，AI 完全不參與判分：`server/src/answers.ts` 的 `checkAnswer()` 做精確比對（支援小數、分數 a/b、帶分數 1 3/8、$、逗號、% 等格式正規化）
- 算術題由生成器產生，`correct_answer` 是程式算的，保證正確；應用題答案存在題庫
- AI 上場時拿到的是後台給的 correct_answer 和 solution sketch（當「引導劇本」用），不需要自己算數學；就算 AI 口誤，也影響不了判分和積分

### C. 防作弊機制（對應需求 5、6）

**要求**：前端傳回後台的 request 只能是「學生提交的答案」；計算積分、判斷對錯、發放獎勵的邏輯必須全部鎖在後台，防止透過瀏覽器 F12 改 JavaScript 變數加分。

**實作**：
- **正確答案從不離開後台**：練習題發題時只回傳題目文字 + token，答案存在 server 記憶體（`pending.ts`）；挑戰賽答案存在 `challenge_runs` 表。前端 JS 裡沒有任何 correct_answer
- **前端只送 `{token, answer}`**（`POST /api/answer`）；對錯判定、第幾次作答、給幾分（首次全額/第二次半額/之後 1 分/看解答 0 分）全在後台算
- **積分只有兩個入口**：後台判分邏輯、家長 PIN 保護的 `/api/admin/points`。不存在任何前端可直接加分的 API
- **兌換防護**：`redeem` 在後台驗餘額、後台扣分、進入待審核；家長批准/退回（退回自動退款）也在後台
- **看解答 = 0 分**在後台記錄（`revealed` flag），先偷看再作答拿不到分
- 挑戰賽每題只能作答一次（後台檢查 `answers[index] !== null`），成績由後台存的作答計算，`finish` 不接受前端傳分數

### D. 本地模型（Ollama）的防護注意事項（對應需求 12、13）

實測發現（2026-07-08）：STRICT SCOPE 提示詞對雲端大模型有效，但 **llama3 8B 守不住離題限制**（會被動漫話題帶走）；**gpt-oss:20b 與 gemma3:27b 都能完美守住**（一句話轉回數學）。因此：
- 本地模式建議用 gpt-oss:20b 或 gemma3:27b（跑 27b 前記得先關 ComfyUI 等吃 VRAM 的程式，否則載入會失敗）
- 本地模式使用時，家長多翻「AI 對話」紀錄
- 判分與積分完全不受影響（AI 無論說什麼都碰不到判分邏輯）

### E. AI 出題的防幻覺管線（對應需求 14）

**背景**：LLM 出題時宣稱的答案可能算錯（尤其本地小模型）；上下文長度在本 app 不是風險（每次 prompt 只有幾 KB 的聚合標籤，見約束 A），**算術幻覺才是**，用架構解：

1. **出題**：依弱項標籤 + 難度 + 主題生成（JSON 結構化輸出），要求單一數字答案
2. **盲解交叉驗證**：另開一個全新上下文的 AI 呼叫，只給題目文字、看不到出題者宣稱的答案，獨立解一次；兩個答案由**程式碼**比對 → `verified` 標記。兩次獨立計算恰好錯成同一個數字的機率極低
3. **家長終審**：所有 AI 生成題一律 `active=0` 進「待審核」區（永遠不會自己上架），後台顯示 ✓已驗證/⚠️未驗證，家長按「上架」才進 Elena 的題目池；⚠️ 的建議自己算過或刪除
4. 上架後的判分照舊走約束 B（程式碼比對），AI 出的題答錯答案也傷不到積分邏輯——最壞情況是一題爛題，且對話紀錄和作答紀錄都能追溯
